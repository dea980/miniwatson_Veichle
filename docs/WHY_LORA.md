# 왜 LoRA 파인튜닝인가 (설계 근거)

> 자동차 도메인 LLM에서 RAG가 이미 도는데 **왜 굳이 파인튜닝을, 그것도 LoRA로** 하는지 정리.
> (JD: "도메인 데이터 기반 LLM 파인튜닝 (PEFT, LoRA 등)")

---

## 1. 파인튜닝 vs RAG — 서로 다른 것을 넣는다

| | RAG | 파인튜닝 |
|---|---|---|
| 넣는 것 | **지식(knowledge)** — 질의 시점에 문서 주입 | **행동/말투/형식/도메인 감각** — 가중치에 새김 |
| 예 | "이 매뉴얼 N페이지 내용은?" | "정비사처럼 단계·주의사항 갖춰 답하기" |
| 갱신 | 인덱스만 갱신 | 재학습 |

RAG로 **못** 바꾸는 것:
- 기본 말투·출력 형식 (일반 챗봇 톤 → 정비 어시스턴트 톤)
- 도메인 용어 유창성 (P0420·프리텐셔너·DTC를 검색 없이 자연스럽게)
- 검색이 빈약하거나 없을 때(오프라인)도 도메인스럽게 답하기

**한 줄: RAG = "무엇을 아는가", 파인튜닝 = "어떻게 답하는가". → 보통 둘 다 쓴다(보완 관계).**

## 2. LoRA vs 풀 파인튜닝 — 왜 LoRA

풀 파인튜닝 = 15억 가중치 전부 재학습 → GPU 없는 환경(M2)에선 비현실(메모리·시간·저장 폭발).

LoRA = 원본 가중치는 동결(freeze), **작은 저랭크 어댑터만** 학습:

| | 풀 파인튜닝 | LoRA |
|---|---|---|
| 학습 파라미터 | 100% (~1.5B) | ~0.1–1% |
| 하드웨어 | GPU 여러 장 | **M2 한 대 가능** |
| 산출물 | 모델 통째(GB) | 어댑터 수~수십 MB |
| 망각(forgetting) | 위험 | 원본 보존 → 안전 |
| 도메인 교체 | 모델마다 새로 | **어댑터만 스왑** |

→ **GPU 없이 도메인 적응을 가능케 하는 표준 기법.** JD가 "PEFT, LoRA"를 콕 집은 이유.

## 3. 왜 "저랭크"가 통하나 (직관)

파인튜닝 시 가중치 변화량 ΔW가 경험적으로 **저차원**이다. d×d 큰 행렬을 통째 바꿀 필요 없이
작은 두 행렬의 곱으로 근사하면 적응 대부분이 잡힌다:

```
출력 = W(동결) · x  +  (A · B)(학습) · x        # B: d×r, A: r×d, r=8~16
```

학습할 숫자가 수백 배 줄어 → 적은 데이터·메모리로 도메인 주입.

## 4. 이 프로젝트에서의 목표 (현실적 기대)

1.5B + 소량 합성데이터 → **GPT를 이기는 게 목표가 아니다.** 보여주는 것:

1. **PEFT/LoRA 수행 역량** (JD 직접 항목)
2. **온디바이스 도메인화** — 작은 모델이 노트북에서 현대차 정비 어시스턴트처럼 답
3. **base vs 파인튜닝 before/after** 정량 비교 (`ml/finetune/eval_adapter.py`)

> 데이터 품질이 9할: 약한 생성 모델로 만든 합성데이터(음역 오류·환각)는 그대로 학습돼 독이 된다.
> → 생성 선생 모델을 키우고(예: qwen2.5:7b) 발췌 근거·고유명사 규칙을 강제한다. 이 판단 자체가 "데이터 기반 파인튜닝" 역량.

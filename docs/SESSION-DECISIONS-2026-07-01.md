## 1. 차종명 표시: 조회하면 이름이 바뀌던 버그

**증상**: 차종 드롭다운은 국내명(싼타페)인데, 조회하면 리포트 히어로가 원본(SANTA FE)으로 떠서 "이름이 바뀌는" 것처럼 보임.

**결정**: 표시 지점(히어로 h2, 케이스 라벨)에 `koModel()`을 적용해 국내명으로 통일. **데이터 키·CarImage·네비게이션 payload는 원본 유지**.

**근거**: 해외명(NHTSA)은 조회·이미지·필터의 조인 키라 바꾸면 안 되고, **표시만** 국내명으로 매핑하는 게 원칙(표시/키 분리). 히어로만 매핑이 빠져 불일치가 났던 것.

## 2. 외국문자(한자·가나) 누수 — 근본원인 2개

**증상**: AI 요약에 `수嗫菲`(Santa Fe 한자 오역)와 `请使用韩语回答…`(시스템 지시가 중국어로) 가 섞여 나옴.

**원인 진단(두 갈래)**:
1. **시스템 프롬프트에 한자·かなカナ 문자가 들어 있었다** → 소형 모델이 그 글자를 그대로 출력에 베낌. "한자·중국어·일본어(かなカナ) 금지"라고 *외국 문자로* 써놓은 게 역효과.
2. **요약에 도메인 FT 1.5B**가 쓰여 영문→한국어 번역 중 고유명사를 한자로 오역. 게다가 깨진 요약이 **캐시**돼 계속 노출.

**결정**:
- 시스템 프롬프트를 **순한글로 재작성**(외국 문자 0). 넣지 않으면 베낄 토큰도 없다.
- 요약은 1회 생성 후 캐시라 반복 비용이 없으니 **항상 강한 기본 채팅 모델** 사용(FT는 도메인 응답 전용으로 역할 분리).
- 생성 후 **CJK 감지 → 1회 재시도 → 그래도 남으면 스크럽**. **오염된 캐시는 읽을 때 무효화**(재생성).
- 프론트 `cleanText`에도 표시단 CJK 제거(최후 방어).

**근거**: 제약은 **생성 단계**(system prompt·모델 선택)에서 거는 게 정공법이고, 사후 스크럽·캐시 무효화는 방어선. "역할별 모델 라우팅"(번역·요약=강한 모델, 도메인 QA=FT)이 핵심 교훈.

## 3. 중요도를 % 로 — 왜 "밴드 지수"인가

**요구**: 중요도를 percentage로 표시.

**함정**: 원점수는 **사망 가중치(×10000)** 때문에 비선형이라, 최댓값 대비 %로 정규화하면 **사망 건만 100%이고 나머지는 0~1%로 뭉개짐**.

**결정**: **절대 심각도 지수 0~100 (밴드 매핑)** — 사망 90~100, 부상 60~89, 화재 40~59, 사고 20~39, 그 외 낮음(`severityPct`, `lib/api.ts`). 정렬은 그대로 원점수(최신성 포함), **표시만** %.

**근거**: 밴드는 (a) 목록이 바뀌어도 값이 안정적이고 (b) "사망 절대 최우선" 철학과 일치. 상대% ·백분위 후보 대비 해석이 쉽고 케이스별 고정값이라 채택.

### 참고 — 왜 중요도 = 사망×10000 + 부상×10 + 화재×5 + 사고×3 ?
정렬 가능한 하나의 숫자로 **"사망 최우선"을 강제**하는 가중합. 사망 ×10000은 1건이 하위 요소가 아무리 쌓여도 못 넘게 벌린 격차(예전 ×100에선 부상40=400이 사망2=200을 역전 → ×10000으로 교정). 사실상 lexicographic 1순위. 부상>화재>사고는 그다음 서열이되 값이 가까워 하위끼리는 **부드러운 혼합**. **정직한 한계**: 경험적 보정이 아닌 휴리스틱 — 원칙이 선 건 사망 격차뿐.

## 4. 추세 차트: 막대 나열 → 꺾은선(면적)

**증상**: 월별·일별 추세를 가로 막대(`Bars`)로 그리니 시간축인데 세로로 줄줄이 쌓여 "나열식"으로 보임.

**결정**: `TrendChart.tsx`(SVG 라인+면적) 신규 — 리콜·불만을 **한 축에 겹쳐** 그리고 hover 시 세로 가이드+각 점+툴팁. y그리드 4단, x라벨 최대 8개로 thinning.

**근거**: 시계열은 선이 추세를 훨씬 잘 보여줌(막대는 순위 비교용). 도넛·바는 구성비/순위에 그대로 두고 추세만 선으로 — **차트는 데이터 형태에 맞춘다**.

## 5. 제품 뼈대 정의 (PRD) — 정비사 wedge

**요구**: "뼈대를 강화하자."

**결정**: [PRD-mechanic-agent.md](PRD-mechanic-agent.md)에 세 축을 명문화 — ① 주 페르소나 하나(**정비사**: 접수→트리아지→진단→견적) ② 측정 가능한 성공지표(근거 인용률 ≥95%, 환각 ≤3%, 치명 오분류 0, CJK 누수 0%, 견적 초안 <3분) ③ 안전 폴백. 나머지(차주·플릿·본사)는 비범위로 내림.

**근거**: 기능은 풍부하나 **주 사용자·성공지표·안전장치라는 "정의"가 부족**했음. 코드가 아니라 정의를 강화하면 기존 기능이 그 밑으로 줄 세워짐. 정비사 wedge는 다단계 Agent 스토리라 두 JD(온디바이스 Agent)와 직결.

**요구**: 근거 없이 단정하는 환각 방지(안전 규격).

## 7. 안전-치명 하드룰 — surfacing 방식 채택

**요구(PRD §7)**: 사망·화재·에어백 미전개 같은 치명 건을 놓치지 않기.

**결정**: `isSafetyCritical(deaths, fire, text)` — 사망·화재 또는 원문의 '에어백 미전개' 신호 감지 → 트리아지·홈에 **⚠ 안전확인 배지**로 surfacing. 중요도 밴드/정렬은 **그대로**.

**근거/트레이드오프(정직)**: PRD 초안은 "치명 건 밴드 90+ 고정"이라 썼지만, 그러면 **화재가 부상을 앞질러** 사용자가 정한 우선순위 모델(부상>화재)과 충돌. 그래서 **순서를 바꾸는 대신 배지로 띄우는** surfacing으로 refine — "놓치지 않기"라는 목적은 달성하면서 기존 서열은 보존.

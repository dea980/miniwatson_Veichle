## 1. 추론 최적화 레버 (무엇을 줄이나)

| 레버 | 효과 | 이 프로젝트 |
|---|---|---|
| **양자화(Quantization)** | 가중치 정밀도↓(16→8/4bit) → 메모리↓·속도↑ | ✅ 메인 (GGUF Q4/Q8, Ollama) |
| KV 캐시 | 이전 토큰 재계산 방지 | Ollama 기본 |
| 출력 길이 제한(num_predict) | 불필요 생성 차단 | 적용(256) |
| 연속 배칭 / PagedAttention | 다중요청 처리량↑ | vLLM(클라우드 GPU) — 배포 단계 |
| 추측 디코딩(speculative) | 작은 모델로 미리 토큰 추측 | 후보 |

> M2 온디바이스의 메인 데모 = **양자화 + 벤치마크**. 배칭류는 GPU 서버(vLLM) 영역.

## 2. 양자화 — 정밀도 vs 비용 트레이드오프

```
fp16(원본)  →  Q8  →  Q5  →  Q4_K_M
크다/정확 ◀──────────────────────▶ 작다/빠르다/약간 부정확
```

정밀도를 낮추면 가중치 1개당 비트수↓ → **메모리 사용·대역폭↓ → 로드·생성 빨라짐.** 대가는 미세한 품질 저하.
"온디바이스"란 결국 **노트북 메모리 안에 모델을 욱여넣고 실용 속도를 내는 것** → 양자화가 핵심.

## 3. 측정 방법 (benchmark.py)

- **TTFT**(Time-To-First-Token): 첫 토큰까지 지연 — 체감 반응성.
- **tok/s**: 초당 생성 토큰 — 처리 속도.
- **메모리**: 로드 크기(`ollama ps`/`ollama list`).
- **품질**: 도메인 golden 몇 개로 정합성(과한 저하 없는지).

```bash
python3 ml/optimize/benchmark.py \
  --models "qwen2.5:1.5b-instruct-q8_0,qwen2.5:1.5b-instruct-q4_K_M" --runs 5
```

## 4. 결과 — Qwen2.5-1.5B 양자화 트레이드오프 (M2)

| 정밀도 | 메모리(로드) | tok/s | TTFT(s) | 품질 |
|---|---|---|---|---|
| fp16 | 3.3 GB | (미측정) | (미측정) | 기준 |
| Q8_0 | 1.8 GB | 4.7 | 0.88 | ~동일 |
| Q4_K_M | ~1.0 GB | **9.5** | **0.62** | 약간↓ |

(Qwen2.5-1.5B, M2, benchmark.py --runs 5, 중앙값)

**해석:** Q4_K_M은 Q8 대비 **메모리 ~1/2(1.8→1.0GB)**, **속도 ~2배(4.7→9.5 tok/s)**, **TTFT 30%↓(0.88→0.62s)**.
도메인 질의 품질 손실은 미미 → **온디바이스 최적은 Q4_K_M.**

> ⚠️ 절대 tok/s가 낮은 건(보통 1.5B는 수십 tok/s) 측정 시 다른 작업(학습 등)이 GPU를 동시 점유했기 때문으로 추정.
> **상대 비교(Q4 vs Q8)는 유효**하나, 깨끗한 절대수치는 단독 실행으로 재측정 권장. (→ §5 직렬화 원칙)

## 5. ⚠️ 자원 관리 — 노트북에선 직렬화(serialize)

단일 M2에서 **생성(7b) + 학습(MLX) + 추론(양자화 여러 개)** 을 동시에 돌리면
통합메모리 초과 → 스왑 → 멈춤. (예: 7b 4.7GB + fp16 3.3GB + q8 1.8GB = ~10GB 동시 점유)

**원칙: 무거운 작업은 한 번에 하나.** 끝나면 메모리 회수 후 다음.

```bash
ollama ps                      # 떠있는 모델 확인
ollama stop <model>            # 안 쓰는 모델 내려 메모리 회수
```

```
[데이터 생성 7b] → stop → [LoRA 학습 MLX] → 종료 → [추론 벤치 1~2개] → stop
```

> **면접 포인트:** 노트북은 직렬화로 버티지만, 운영에선 이래서 **학습/서빙/추론을 별도 노드·서비스로 분리**한다.
> 이 프로젝트가 ML 사이드카(Python)와 서빙(Java)을 나눈 이유와 정확히 연결된다. (→ VEHICLE_ARCHITECTURE.md)

## 6. 온디바이스 vs 클라우드 (대응 전략)

| | 온디바이스 | 클라우드 |
|---|---|---|
| 서빙 | Ollama + GGUF Q4 (이 데모) | vLLM(OpenAI 호환) + AWQ/배칭 |
| 모델 연결 | `llm.provider=ollama` | `llm.provider=vllm`(예정) |
| 장점 | 프라이버시·오프라인·무비용 | 처리량·확장 |

→ `LlmClient` 추상화로 **설정만 바꿔 전환** = JD "온디바이스/클라우드 환경 대응".

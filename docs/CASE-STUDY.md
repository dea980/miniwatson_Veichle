# MiniWatson — Case Study (인터뷰용 정리)

면접에서 "이 프로젝트 설명해봐" / "가장 어려웠던 문제는?" / "왜 그렇게 했어?"에 바로 답하기 위한 한 장 정리.
긴 README가 아니라, **말로 풀 수 있는 narrative**가 목적이다. 위에서 아래로 읽으면 5분 발표 대본이 된다.

---

## 1. 한 줄 (elevator)

> IBM watsonx의 3계층(data, ai, governance)을 Spring Boot로 처음부터 다시 만든 엔터프라이즈 RAG 플랫폼.
> "엔터프라이즈 GenAI는 모델이 아니라 파이프라인, 저장포맷, 감사추적에서 어렵다"를 **직접 만들어서** 확인한 프로젝트.

면접관이 "watsonx가 뭔지 아냐"고 물으면 이 한 줄이 곧 답이다. **읽은 게 아니라 만들어봤다**가 차별점.

## 2. watsonx 3계층 매핑 (이게 뼈대다)

| watsonx | MiniWatson 구현 | 핵심 책임 |
|---|---|---|
| **watsonx.data** | Parquet 콜드스토어 + H2 카탈로그, 티어드 저장(JSON hot → Parquet cold), 멀티테넌트 namespace | KB가 진실의 원천. 벡터인덱스/pgvector는 여기서 재구축되는 파생물 |
| **watsonx.ai** | Ollama 멀티 LLM, 768차원 임베딩, 하이브리드 검색(벡터+BM25, RRF), 2단계 리랭킹, 비전+OCR, DuckDB text-to-SQL | 검색·생성. 전략패턴으로 청커/벡터스토어/리랭커/인증을 전부 교체 가능 |
| **watsonx.governance** | 전 LLM 호출 감사로그, PII 마스킹(저장 전), provenance(답변별 출처 청크), 테넌트 격리 강제 | 누가·언제·무엇으로 답했나를 추적. 거버넌스가 가용성을 죽이지 않게 fail-open |

발표 흐름: **"데이터를 어떻게 넣고(data) → 어떻게 찾아서 답하고(ai) → 그걸 어떻게 감사하나(governance)"** 순서로 말하면 청중이 따라온다.

## 3. 핵심 기술 결정 (왜 그렇게 했나)

면접에서 "왜 X를 골랐어?"는 거의 무조건 나온다. 결정마다 **버린 대안**을 같이 말해야 설득된다.

**전략패턴 + `@ConditionalOnProperty`** — 청커(fixed/recursive/semantic), 벡터스토어(인메모리/pgvector), 리랭커(none/llm/mmr/cross), 인증(A/B/C)을 전부 설정 한 줄로 교체. *왜?* 학습 프로젝트라 "어느 게 더 나은가"를 측정하려면 갈아끼울 수 있어야 했다. 운영에서도 같은 코드가 환경별로 다르게 뜬다(sovereignty).

**임베딩 모델 4종 비교 → granite-embedding:278m** — nomic / 30m / mxbai를 골든셋으로 돌려 recall 측정. 278m이 다국어(한국어 11/11) + recall 97%로 승. *왜 측정?* "큰 모델이 낫겠지"는 가설일 뿐. 768차원 다국어가 우리 코퍼스에 맞았다. (EMBEDDINGS.md)

**하이브리드 검색(벡터+BM25, RRF)** — 임베딩은 "INV-2026-0042" 같은 정확토큰을 못 잡는다. BM25(렉시컬)가 잡는다. 점수정규화 없이 순위기반 RRF로 융합. *솔직한 한계도 말한다*: 작고 깨끗한 코퍼스에선 이득이 작고, 크고 노이즈 많은 데이터에서 빛난다.

**pgvector 영속화** — 인메모리는 차원실험용, 운영은 pgvector(HNSW). vector(768) 고정, namespace 필터로 테넌트 격리. 인메모리와 **35/35 동일 recall** 달성이 검증 기준이었다.

**멀티테넌트 보안 A/B/C** — API key 필터 / Spring Security+API key / JWT 세 방식을 다 만들고 장단점 비교. 공통은 authN(누구냐)과 authZ(접근권한)를 분리: 인증 어댑터가 TenantContext를 채우고, TenantGuard가 강제. (SECURITY.md)

## 4. 가장 어려웠던 문제 (면접 골드 — "어려운 버그?"의 답)

이 절이 제일 중요하다. 면접관은 결과보다 **디버깅 사고과정**을 본다. 각 버그를 "증상 → 가설 제거 → 진짜 원인 → 교훈"으로 말하면 시니어처럼 들린다.

### 4.1 pgvector recall이 26/35, 인메모리는 35/35 (id 붕괴)
- **증상**: 같은 코퍼스인데 pgvector만 9개 덜 맞음.
- **제거한 가설**: ANN 근사오차(EXPLAIN 찍어보니 exact scan), float4 정밀도(double 재정렬해도 동일), 직렬화 손실(round-trip 무손실), 좀비 JVM.
- **진짜 원인**: RRF 융합이 `Article.id`로 후보를 키잉하는데, pgvector에서 재구성한 Article의 id가 primitive long 기본값 0이라 **전부 한 키로 붕괴**.
- **해결**: 스키마에 `article_id` 컬럼 추가, 저장·복원. → 35/35.
- **교훈**: 분산/재구성 경로에선 **식별자 보존**이 정답 키. 모델 탓이 아니라 객체 재구성 탓이었다.

### 4.2 CI에서만 컴파일 실패 ("cannot find symbol VectorStore")
- **증상**: 로컬은 되는데 CI만 깨짐.
- **원인**: `.gitignore`의 `data/`(앵커 없음)가 **소스 패키지 `com/miniwatson/data/`까지** 무시 → 소스가 커밋된 적이 없음.
- **해결**: `/data/`로 앵커. `git ls-files | grep VectorStore`로 추적 확인.
- **교훈**: "로컬은 되는데 CI가 깨진다"는 거의 **환경 차이 = 무엇이 실제로 커밋됐나**. gitignore 패턴은 앵커 안 하면 어디서든 매칭된다.

### 4.3 좀비 JVM (몇 시간 날림)
- **증상**: 코드를 고쳐도 옛 동작이 8080에서 계속 나옴.
- **원인**: `pkill -f spring-boot`가 **fork된 앱 JVM**을 못 죽여서 구버전이 계속 서빙.
- **해결**: `pkill -9 -f java` + `jps`/`lsof`로 확인. 부팅 로그에 BUILD-MARKER 찍어 "새 코드 맞나"를 증명.
- **교훈**: 디버깅 전에 **내가 고친 코드가 진짜 돌고 있나부터 증명**한다.

### 4.4 비전 모델이 송장 숫자를 지어냄
- **증상**: llava가 페이지에 없는 invoice total을 발명.
- **해결**: 모델을 키운 게 아니라 **역할 분리** — OCR(Tesseract)=정확한 텍스트/숫자, 비전=레이아웃/맥락, 프롬프트에 "충돌 시 OCR을 신뢰하라" 명시.
- **교훈**: 소스를 합치는 것만으론 부족하다. **어느 소스가 authoritative인지 선언**해야 한다.

(전체 목록은 DEBUGGING.md. 위 4개는 발표용으로 고른 대표 사례.)

## 5. 측정된 결과 (숫자로 말한다)

- 검색 recall **35/35** (인메모리 = pgvector 패리티), 임베딩 비교 승자 recall **97%**, 한국어 **11/11**.
- Parquet(SNAPPY) 저장이 JSON 대비 **7배** 작음 (54KB → 7.8KB).
- gemma `think:false`로 지연 **약 3배** 단축.
- **53개 단위테스트** = CI 게이트, 외부 의존 0. GitHub Actions + GitLab CI 양쪽 초록, 멀티아치(amd64+arm64) 이미지 GHCR 푸시.
- 보안 A/C 라이브 검증: 401(무인증) / 200(정상) / 403(타테넌트) 흐름 확인.

## 6. 다음 (로드맵 — "더 했으면?"의 답)

- **라이브 배포** — 배포 산출물(prod compose, 멀티아치 CI, Oracle Always Free 가이드)은 준비됨. 무료 ARM CPU 추론이 느려서 라이브는 보류, 로컬 데모로 시연.
- **보안 Tier 2** — 프롬프트 인젝션 방어(시스템 프롬프트 경계), PII 커버리지 확대, TLS/레이트리밋.
- **평가 심화** — RAGAS류 답변품질(검색 recall만이 아니라 생성 정확도), 관측성(metrics/health/tracing).

## 7. 한 문장 결론 (마무리 멘트)

> "모델 고르기는 쉬운 부분이었다. 어려운 건 파이프라인, 저장포맷, 감사추적 — 즉 엔터프라이즈가 실제로 돈 쓰는 곳이다.
> 그걸 읽은 게 아니라 만들어서 알게 됐다."

이건 IBV CEO Study의 "6개 역량으로 5.4배 채택"(체인지매니지먼트, AI거버넌스, 데이터거버넌스, 실시간통합, 시스템통합, 재무통합)과도 맞물린다. 컨설팅 면접에서 비즈니스 언어로 연결하기 좋은 고리.

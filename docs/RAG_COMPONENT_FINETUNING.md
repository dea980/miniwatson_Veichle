# RAG 컴포넌트별 파인튜닝 전략

> "chat 모델만 파인튜닝하나? 다른 모델은?" → RAG 파이프라인의 **각 단계**를 도메인 적응할 수 있다.
> 어디를, 왜, 어떻게, 그리고 이 프로젝트의 단계별 우선순위.
> 관련: [WHY_LORA.md](WHY_LORA.md), [PEFT_METHODS.md](PEFT_METHODS.md)

---

## 1. 파이프라인의 어느 단계든 적응 가능

```
질문 → [임베딩] ── 검색 정확도        ← contrastive 파인튜닝
     → [BM25]
     → [리랭커] ── top-K 순위          ← cross-encoder 파인튜닝
     → [생성 LLM] ── 답변 말투와 도메인   ← LoRA (이 프로젝트 완료분)
     → 답변 + 근거
```

핵심: **생성 파인튜닝 = "어떻게 답하나", 임베딩 파인튜닝 = "올바른 근거를 찾나".** RAG 품질은 두 축 모두에 달려있다.

## 2. 컴포넌트별 정리

| 컴포넌트 | 모델 | 방법 | 학습 데이터 | 효과 | 우선순위 |
|---|---|---|---|---|---|
| **생성(Chat)** | Qwen2.5-1.5B | LoRA(SFT) | (instruction, 답변) 쌍 | 답변 말투, 용어, 형식 | ✅ 완료 |
| **임베딩** | granite-embedding 278m | Contrastive(대조학습) | (질문, 정답청크) [+오답] | **검색 recall↑** (엉뚱한 청크 방지) | 🔥 다음 레버 |
| **리랭커** | BGE cross-encoder | (query, doc, label) 분류 | 관련/비관련 쌍 | top-N→K 순위 정밀화 | 보조 |
| **비전** | llava / granite-vision | LoRA(멀티모달) | (이미지, Q&A) | 계기판 경고등과 부품 인식 | 니치 |

## 3. 왜 "임베딩 파인튜닝"이 RAG의 핵심 레버인가

증상: "브레이크 경고등" 질문에 EDR이나 안전벨트 청크가 잡힘(=검색 미스). 생성 모델을 아무리 튜닝해도, **틀린 근거를 받으면 답도 틀린다.**

→ 임베딩을 도메인 데이터로 **대조학습**하면 "질문 ↔ 정답 청크"를 임베딩 공간에서 가깝게 당겨 검색 정확도가 오른다.

방법 개요:
1. **데이터**: `(질문, 관련 청크)` 양성쌍 + (가능하면) 하드 네거티브. → `build_dataset.py`와 유사하게, 청크별로 "이 청크가 답이 되는 질문"을 생성해 페어 구성.
2. **학습**: `sentence-transformers`의 MultipleNegativesRankingLoss 등. base = granite-embedding(278m). M2에서 소형이라 가능.
3. **검증**: 도메인 golden recall이 base 임베딩 대비 오르는지(`eval/run_eval.py`).

> 주의: 임베딩을 바꾸면 **기존 인덱스를 재임베딩**해야 한다(차원/공간이 달라짐). 그래서 별도 작업 + 재인제스트가 필요.

## 4. 이 프로젝트의 단계별 우선순위

1. **[완료/진행] 생성 LoRA** — 답변 말투와 도메인 (JD 헤드라인, base vs FT 비교)
2. **[다음] 임베딩 파인튜닝** — RAG 검색 recall이 부족할 때. 가장 큰 RAG 품질 레버.
3. **[보조] 리랭커 파인튜닝** — 순위 미세조정.
4. **[니치] 비전 LoRA** — 자동차 이미지(경고등) 도메인.

## 5. 면접 한 줄

> "RAG는 생성, 검색, 재순위로 나뉘고, 각 단계를 도메인 적응할 수 있다. 우선 **생성을 LoRA**로 튜닝했고, 검색이 병목이면 **임베딩을 대조학습**으로, 그 다음 리랭커 순으로 단계적으로 최적화한다. 즉 '한 모델'이 아니라 **파이프라인 전체를 도메인화**하는 관점."

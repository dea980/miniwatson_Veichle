# A/S 운영 기능 — 차종/케이스 2레벨, 트리아지, 점검 체크리스트, 정비 스케줄

리콜·불만·매뉴얼 데이터를 **정비/AS 현장 업무**로 잇는 기능 묶음이다. 핵심 설계 원칙은 **차종(모델 집계) 레벨과 케이스(개별 접수 건) 레벨을 분리**하고, 화면을 그 두 레벨에 맞춰 배치한 것이다.

## 1. 2레벨 정보 구조

| 레벨 | 단위 | 화면 | 질문 |
|---|---|---|---|
| **차종** | 모델 전체 집계 (예: PALISADE) | 분석 대시보드, 차종 진단 리포트 | "이 차종은 전반적으로 어떤가?" |
| **케이스(건)** | 개별 접수 건 (접수번호=ODI) | 케이스 트리아지, 리포트 내 "이 차종의 케이스" | "이 접수 건은 무엇이고 어떻게 대응하나?" |

초기에는 둘이 뒤섞여 있었다("차량 진단 리포트"가 사실 모델 집계였음). 이를 **차종=집계 / 케이스=개별**로 명확히 갈라 이름·화면·데이터 경로를 정리했다.

## 2. 케이스 우선순위 트리아지

고객 불만(접수)을 **심각도 우선순위**로 정렬해 "먼저 대응할 건"을 위로 올린다.

```
우선순위 = 사망×100 + 부상×10 + 화재×5 + 사고×3   (+ 동점 시 최신순)
```

- 전 차종 횡단(`/api/analytics/cases`) — 차종·부위 키워드 필터.
- 각 행: 접수번호·차종·연식·접수일 + 심각도 배지(사망/부상/화재/사고) + 좌측 강조선(사망·화재=빨강).
- 건별 "차량 진단" → 그 건의 필요 부품(견적) 인라인.

집계는 **결정적 SQL(DuckDB)** 로, 우선순위 점수도 SQL에서 계산한다(LLM 아님 → 환각 차단).

## 3. 건별 점검 체크리스트

차종 집계가 아니라 **개별 접수 건에 맞춘** 점검 항목을 만든다.

```
공통 점검   : 성능·상태점검기록부 표준 8종 (원동기/변속기/.../전기/외판·골격) — 모든 차량
이 건 점검  : 그 건의 결함 부위(components) → 한국어 점검 항목   (예: AIR BAGS → 에어백 경고등·전개 점검)
```

- 매핑표: `data/vehicle/inspection_map.csv` (`keyword → 점검항목`, 부분일치). 없으면 코드 내 빌트인 폴백.
- 엔드포인트: `GET /api/analytics/checklist?model=&component=` — `component` 주면 **건별**(그 부위만), 없으면 차종 집계(부위 빈도순).
- 발견: 표준 점검표는 *기계 계통*만 다루고, 리콜·불만은 *안전·전자·ADAS*(안전벨트·에어백·전방충돌·후방카메라)를 드러낸다 → 그 빈틈이 곧 "이 차/이 건에 추가로 봐야 할 것".

## 4. 정비 스케줄 (캘린더)

달력으로 정비 일정을 예약·관리하고 상태(예정/진행/완료)를 추적한다.

- **새 DB 없음** — 기존 JPA 데이터소스(H2 로컬 / PostgreSQL 운영)에 `maintenance_schedule` 테이블만 추가(`ddl-auto: update`로 자동 생성). 감사 로그·카탈로그와 같은 DB를 공유한다.
- 엔티티 `MaintenanceSchedule`(model·caseNumber·title·scheduledDate·status·technician·note) + `MaintenanceRepository` + `MaintenanceController`(목록/추가/상태변경/삭제).
- 접수번호(caseNumber)로 트리아지 케이스와 연결 가능 → "우선순위 케이스 → 정비 예약" 흐름.

## 5. 차종 진단 리포트 (개편)

단일 차종 심화 화면. 입력을 단순화하고 케이스·체크리스트를 한 흐름으로 묶었다.

- namespace 입력 제거(내부 `vehicle` 고정), 차종은 **실제 차종 드롭다운**(불만 데이터 기준), LLM 모델만 작게 노출.
- 구성: 차종 사진 배너 → KPI → 점검표 → 리콜/불만 부품 → 종합 진단서(LLM) → **이 차종의 케이스**(부위 검색 + 건별 점검·부품).

## 6. 보조 — 대화형 어시스턴트 · 차종 사진 · 지식베이스 정리

- **멀티턴 RAG 채팅**(홈) — 말풍선 스레드 + 메시지별 출처. 모델 선택(지시를 잘 따르는 instruct 우선; 약한 모델은 프롬프트를 메아리치는 퇴화가 있어 기본값을 분리).
- **차종 사진** — 위키백과 REST 요약 API에서 브라우저가 직접(CORS 허용) 가져오고, 없으면 차 실루엣 SVG로 폴백. 홈 히어로·차종 카드·리포트 헤더에 표시.
- **지식베이스** — 문서를 카테고리(매뉴얼/진단리포트/기타) → 차종으로 그룹핑(제목 기반 자동 분류).

## 7. 데이터·동시성 메모

- 모든 집계·트리아지·체크리스트는 DuckDB(`TabularSqlService`)의 **결정적 SQL**. LLM은 서술만.
- 대시보드/리포트가 여러 쿼리를 병렬로 쏘면서 드러난 **DuckDB 단일 커넥션 동시성 문제**는 직렬화 + 자가복구 + 등록 1회로 방어했다 — 근거 [TABULAR-SQL.md](TABULAR-SQL.md) §4.5, 사건 [DEBUGGING.md](DEBUGGING.md) §11.3.

## 8. JD 매핑

- **NLP/LLM 직무**: RAG·Agent·도메인 LoRA 위에, 결정적 SQL + LLM 서술의 하이브리드(환각 차단).
- **데이터 분석 직무**: 우선순위 트리아지(클레임 분류), 부품 수요·워런티 비용 프록시, 점검 체크리스트(품질 신호→현장 액션) — "분석에 LLM을 적용"하는 구체 사례.

## 9. 로드맵 — 커넥티드카 개인화 (JD #2, 미구현)

차량별 정비이력 + 운전 스타일로 **개인화 진단·예측 정비·맞춤 견적**을 하는 방향. JD #2(커넥티드카 ECU·워런티·예측)에 직결.

**무엇으로 무엇을 추론하나 (정직하게):**

| 데이터 | 추론 | 비고 |
|---|---|---|
| 정비/클레임 이력(보유 유사) | 고장 패턴·부품 마모·리콜 노출 | **운전 스타일은 직접 추론 불가** |
| **텔레매틱스/ECU**(미보유) | 운전 스타일(급가속·급제동), 예측 정비 | 딜러 독점 데이터 — 데모는 mock 필요 |

핵심: **운전 스타일은 정비 기록만으론 안 되고 텔레매틱스/ECU 센서 데이터가 필요**하다(브레이크 마모율 같은 간접 신호는 약함). 그래서 이 기능은 *데이터 요건이 충족될 때(또는 mock으로)* 붙이는 로드맵 레이어이며, MVP 스파인(매뉴얼 RAG · 리콜 분석 · 케이스 진단)이 아니다. 데모로 보여주려면 *mock 텔레매틱스 요약 1건 → 개인화 권고*("주행거리 대비 급제동↑ → 브레이크 점검 주기 단축") 정도가 적정 범위.

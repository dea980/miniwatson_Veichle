# Cloud Deployment — 벤더 중립 배포 (Phase 0)

취업용 한시 데모(3~6개월)를 클라우드에 올리는 계획. 핵심 원칙 하나로 시작한다.

> **추론 제공자와 컴퓨트 호스트는 "코드가 묶이는 결정"이 아니라 "배포 시점에 고르는 설정값"이다.** 그래서 먼저 추론 호출을 인터페이스로 추상화하고, 제공자/호스트는 그때 무료로 쓸 수 있는 걸로 고른다. 이러면 벤더를 바꿔도 코드와 문서를 갈아엎지 않는다(락인 회피).

상위 결정 근거 정리, 로컬 셋업은 [DEPLOYMENT.md](DEPLOYMENT.md), 임베딩 근거는 [EMBEDDINGS.md](EMBEDDINGS.md), 보안은 [SECURITY.md](SECURITY.md), 운영은 [OPERATIONS.md](OPERATIONS.md).

## 0. 사전 점검 — 배포 전에 반드시 (OpenJ9 크래시)

루트에 `javacore.*`, `Snap.*`, `jitdump.*` 덤프가 쌓여 있다. 현재 Dockerfile이 `ibm-semeru-runtimes:open-21-jre`(OpenJ9)를 쓰는데, 이 런타임에서 앱이 죽고 있다는 신호다. **이걸 먼저 잡지 않으면 클라우드에 올려도 같은 크래시가 난다.**

| 점검 | 작업 | 검증 |
|---|---|---|
| 크래시 재현/원인 | `javacore.*` 최신 파일에서 스택과 사유(메모리/SecurityManager/JIT) 확인 | 원인 한 줄로 특정 |
| 런타임 후보 비교 | OpenJ9 유지 vs HotSpot(`eclipse-temurin:21-jre`)로 베이스 교체 시도 (`run-hotspot.log`에 시도 흔적 있음) | 동일 부하에서 크래시 재발 여부 |
| 결정 | 안정적인 런타임으로 Dockerfile 베이스 확정 | 로컬에서 데모 시나리오 완주, 덤프 미발생 |

배포 가능 판단의 1순위는 동시성이 아니라 이 JVM 안정성이다. 여기부터 통과시킨다.

## 1. 두 결정을 분리한다

| 결정 | 무엇 | 교체 가능성 |
|---|---|---|
| 추론 제공자 | LLM 생성/비전/임베딩을 어디서 | 추상화하면 설정으로 교체 |
| 컴퓨트 호스트 | 앱 + pgvector를 어디서 실행 | docker-compose라 어느 VM이든 |

둘은 독립이다. 그래서 벤더를 지금 확정하지 않아도 된다.

## 2. LlmClient 추상화 (이게 진짜 작업, 제공자 무관)

**왜**: 지금 `application.yaml`의 `ollama.*`를 직접 호출하는 커스텀 HTTP 클라이언트를 쓴다. 클라우드를 바꿀 때마다 호출부를 갈아엎으면 매번 코드와 문서가 뒤집힌다. 인터페이스를 한 번 두면 그 비용을 0으로 만든다. 이 추상화 자체가 "벤더 락인 회피"라는 아키텍처 어필 포인트다.

**작업**:
- `LlmClient` 인터페이스 정의: `chat()`, `vision()`, `embed()`
- 기존 호출부를 `OllamaClient` 구현체로 이동 (동작 그대로)
- 설정 키 `llm.provider`로 구현체 선택 (`ollama` | `watsonx` | `bedrock` | `vertex`)
- 제공자별 구현체는 해당 제공자를 고를 때만 추가

**검증**: `llm.provider=ollama`로 로컬에서 기존 동작 유지.
```bash
python3 eval/run_eval.py     # recall이 추상화 전과 동일해야 함(회귀 0)
```
[EVALUATION.md](EVALUATION.md) golden set 기준 recall 불변이면 통과.

## 3. 추론 제공자 백엔드 (필요할 때 구현, 골라서)

| 제공자 | 생성/비전 | 임베딩 / 차원 | 무료 조건 | 코드 변경 | 현재 상태 |
|---|---|---|---|---|---|
| Ollama(자체호스팅) | granite4 / llava | granite-embedding 278m / **768** | 호스팅 비용만 | 0 (현재) | 동시성 약함(CPU) |
| watsonx.ai | granite / granite-vision | granite-embedding-278m-multilingual / **768** | $200 + Lite(월 10 CUH) | 최소(차원 동일) | granite 네이티브, 테마 일치 |
| AWS Bedrock | Claude(비전 포함) | Titan v2 / **1024** | 신규 $200/6개월 | 중(재인덱싱) | 신규 자격 막힘 |
| GCP Vertex | Gemini 2.5 Flash | gemini-embedding-001@768 / **768** | 신규 $300/90일 | 중(모델 교체) | 자격 불확실(이전 사용) |

**선택 기준**: (1) 지금 실제로 무료로 쓸 수 있는가, (2) 코드 변경이 적은가(임베딩 768 유지면 재인덱싱 없음), (3) 취업 서사에 맞는가. 추상화가 돼 있으니 처음 고른 게 마음에 안 들면 설정만 바꿔 교체한다.

임베딩 차원이 768로 유지되는 제공자(watsonx, Vertex@768)면 pgvector 컬럼/`article.avsc` 변경 없이 **재임베딩만** 하면 된다. Bedrock(1024)만 스키마 변경 + 재인덱싱이 따른다.

## 4. 컴퓨트 호스트 (어디든 docker-compose)

| 호스트 | 사양/비용 | 비고 |
|---|---|---|
| Oracle Always Free | 2 vCPU / 12GB 무료(2026-06-15 축소) | 추론을 밖으로 빼면 충분. ARM이라 멀티아치 필요 |
| Hetzner 정액 VPS | 약 €16/월(8 vCPU/16GB, x86) | 정액이라 과금 폭탄 없음, 가성비 1위 |
| GCP Compute Engine | e2-medium 약 $25/월(크레딧 내) | 크레딧 받으면 90일 무료 |
| 자체 VM/온프렘 | 가변 | Ollama 자체호스팅 유지 시 |

추론을 관리형으로 빼면 호스트는 앱+pgvector만 돌리므로 작은 박스로 충분하다. JVM 힙(`-Xmx`)을 묶어 OOM을 피한다.

## 5. 동시성과 병목, 그리고 가드

**병목은 LLM 추론 한 곳이다.** pgvector 검색, JVM, VM은 데모 규모에서 막히지 않는다.

- 자체호스팅 Ollama(CPU): 동시 요청이 CPU를 두고 경합 → 사실상 직렬, 느림. 동시성 최악.
- 관리형(watsonx/Bedrock/Vertex): 동시성은 벤더가 처리. 대신 무료 한도/레이트리밋이 천장(watsonx Lite 월 10 CUH 등).

현실: 포트폴리오 데모의 동시 부하는 낮고 산발적이라(면접관 1~3명, 가끔) 병목이 실제로 터질 일은 드물다. 다만 터졌을 때 망가져 보이지 않게 싼 가드를 깐다.

| 가드 | 왜 | 검증 |
|---|---|---|
| 요청 타임아웃 + "생성 중" UI | 멈춘 것처럼 안 보이게 | 느린 응답에서 UI가 상태 표시 |
| 동시 처리 수 제한(세마포어/큐) | 폭주 시 줄세우기 | 동시 N+1 요청 시 초과분 대기/거절 |
| IP/API키 레이트리밋 | 한 명이 토큰/크레딧 태우는 것 방지 | 임계 초과 시 429 |
| 쿼터/비용 모니터 | 무료 한도 초과 사전 감지 | 한도 근접 시 알림 |

관리형 추론을 택하면 동시성 스케일링을 벤더에 위임하게 된다. 이것도 관리형 + 추상화를 고르는 근거다.

## 6. 배포 단계 (제공자/호스트 정해지면 공통)

1. **(자체호스팅이 아니면) 제공자 클라이언트 구현 + 인증** — 키 파일/액세스 키를 디스크에 두지 않는다(인스턴스 역할/서비스 계정/ADC). 검증: 자격증명이 역할 기반으로 잡히는지.
2. **임베딩 재생성** — 모델이 바뀌면 재임베딩. 차원이 바뀌면 pgvector 컬럼도 변경. 검증: `/api/data/index/stats`의 dim/count, `eval/run_eval.py` recall.
3. **compose 조정** — 관리형으로 빼면 `ollama`/`ollama-init`/`OLLAMA_URL` 제거. 검증: `docker compose ps`에 불필요 서비스 없음.
4. **VM 프로비저닝 + 기동** — 방화벽에서 5432/11434 외부 비노출, 22는 내 IP만. 검증: 포트 노출 점검, `/actuator/health` UP.
5. **HTTPS + 인증** — Caddy/Nginx + Let's Encrypt, `SECURITY_ENABLED=true`. 검증: 유효 인증서, 무인증 호출 401.
6. **비용 가드** — 예산 알림 + 종료/소진 리마인더. 검증: 알림 규칙 활성, 일일 비용 예상 범위.

## 8. 보안 / 데이터 / 거버넌스

- 인증: 제공자 호출은 역할/서비스계정으로, 정적 키 없음, 최소 권한.
- 시크릿: `.env`만, 리포엔 `.env.example`만.
- 데이터 경계: 관리형 추론이면 질의/문서 일부가 제공자로 나간다. 데모 코퍼스는 공개 가능한 샘플로 한정, 민감 데이터 제외. 근거는 [DEEP-DIVE-DATA-GOVERNANCE.md](DEEP-DIVE-DATA-GOVERNANCE.md).
- 감사: governance 로그로 질의/응답 추적. Phase 3에서 모델/설정 메타까지 묶는다.
- 파기: 데모 종료 시 인스턴스/볼륨/데이터 삭제 절차 사전 정의.

## 9. 다음 단계 (Phase 1~3 로드맵)

- **Phase 1 CI/CD**: `.github/workflows`로 빌드 → 테스트 → eval 게이트(recall 임계 미달 시 배포 차단) → 이미지 푸시 → 배포 자동화.
- **Phase 2 운영 모니터링**: 지연, 토큰 비용, 검색 품질(recall) 추적. governance 로그를 시계열로 보고 드리프트 감지.
- **Phase 3 모델 버전 레지스트리**: 어떤 모델/설정 조합으로 어떤 응답이 나갔는지 추적. 응답에 모델 버전과 설정 해시 기록.

각 Phase는 별도 문서로 같은 형식(왜/작업/검증)으로 작성한다.

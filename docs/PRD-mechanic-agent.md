# PRD — 정비사 AI 어시스턴트 (접수 → 트리아지 → 진단 → 견적)

> **목적**: 지금까지 사방으로 퍼진 기능들을 하나의 주 사용자와 핵심 Job 아래로 줄 세워 "제품의 뼈대"를 세운다.
> 이 문서가 정의하는 세 축 — **① 주 페르소나 하나 ② 측정 가능한 성공지표 ③ 안전·신뢰 폴백** — 이 곧 뼈대다.
> 나머지 기능(차주 셀프서비스, 플릿 분석, 텔레매틱스 개인화)은 이 뼈대 밑의 로드맵으로 내린다.

## 1. 문제 정의

카센터·서비스센터 정비사(또는 서비스 어드바이저)는 새 접수 건이 들어올 때마다 세 가지에서 시간을 쓰고 실수 위험을 진다.

1. **우선순위 판단** — 오늘 들어온 건 중 무엇을 먼저 봐야 하나. 안전이 걸린 건(사망·화재·에어백 미전개)을 놓치면 치명적인데, 판단이 사람마다 다르고 일관성이 없다.
2. **원인·근거 찾기** — 증상에서 결함 부위를 추정하고 매뉴얼 근거를 확인하는 데, 방대한 오너스/정비 자료를 사람이 뒤진다. 경험 편차가 크다.
3. **부품·견적 산정** — 필요 부품과 공임을 뽑아 고객에게 설명하는 초안 작성에 시간이 든다.

지금 앱은 이 세 가지 조각(트리아지·건별 체크리스트·부품 견적)을 **이미 다 갖고 있지만**, "정비사가 한 접수 건을 처음부터 끝까지 몰고 가는 하나의 흐름"으로 묶여 있지 않고, 품질이 측정되지 않으며, 틀렸을 때의 안전장치가 없다.

## 2. 주 페르소나 (하나만)

**정비사 / 서비스 어드바이저** — 하루 수십 건 접수를 처리. 안전 위험 건을 절대 놓치면 안 됨. 매뉴얼·부품 지식에 개인 편차. 화면 앞에서 고객과 대화하며 빠른 근거가 필요.

명시적 비대상(이번 릴리스): **차주(셀프서비스)**, **플릿 운영자/분석가**, **본사 품질팀**. 이들은 신뢰 기준·톤·화면이 달라 별도 wedge로 분리한다.

## 3. 핵심 Job (JTBD)

> "새 접수 건이 들어오면, **3분 안에** 중요도를 판단하고 → 결함 부위를 매뉴얼 근거와 함께 진단하고 → 필요 부품·견적 초안까지 뽑아 고객에게 설명한다. 확실하지 않으면 어시스턴트가 스스로 '확실치 않음'을 말한다."

## 4. 범위

| | 포함 (In) | 비포함 (Out — 이번 릴리스) |
|---|---|---|
| 기능 | 트리아지(중요도), 건별 점검 체크리스트, 매뉴얼 근거 진단, 부품·견적 초안, 정비 스케줄 연결 | 차주 셀프 Q&A, 플릿 분석 대시보드, 텔레매틱스/ECU 개인화 |
| Agent | 접수→트리아지→진단→견적 다단계 오케스트레이션 + 트레이스 | 완전 자율 실행(정비사 확인 없는 발주 등) |
| 배포 | 로컬/사내 데모, 캐시로 반복 질의 가속 | 실차 ECU 온디바이스(예산만 명시, 구현 X) |

## 5. 성공지표 (측정 가능해야 뼈대다)

"측정 없이 개선 없다." 아래는 릴리스 게이트로 삼는 목표치다. 현재 eval 하네스([EVALUATION.md](EVALUATION.md))는 플랫폼 검색 recall 중심이므로, **정비사 wedge 전용 골든셋과 안전 지표를 추가**한다.

| 축 | 지표 | 목표 | 측정 방법 |
|---|---|---|---|
| 정확도 | 답변 근거 인용률 | ≥ 95% | 골든셋 — 답변에 최소 1개 유효 source |
| 정확도 | 환각률(근거 없는 단정) | ≤ 3% | refusal 케이스 + faithfulness 채점 |
| 안전 | 트리아지 치명 건(사망·화재) 오분류 | **0** | 라벨된 케이스에서 하드룰 검증 |
| 진단 | 결함 부위 top-1 정확도 | ≥ 80% | 접수↔부위 라벨 골든셋 |
| 품질 | 외국문자(한자·가나) 누수율 | **0%** | 출력 CJK 스캔(회귀 테스트) |
| 효율 | 접수→견적 초안 소요(중위값) | < 3분 | 세션 계측 |
| 성능 | RAG p95(콜드) / 캐시 히트 | 목표치 / < 100ms | k6 ([GUIDE-answer-cache.md](GUIDE-answer-cache.md)) |
| 신뢰 | 정비사 👍 비율 | ≥ 목표 | 피드백 로그 |

## 6. 품질 / eval 하네스 (CI 게이트)

기존 `eval/golden.json`을 확장한다.

- **wedge 골든셋 추가**: 실제 유형의 접수 케이스 N개 + 기대값(중요도 밴드, 결함 부위, 필요 부품군, 기대 근거 문서).
- **안전 케이스**: 근거 없는 질문 → "모름" 폴백 확인(confident 오답 금지), 치명 건 하드룰.
- **회귀 스캔**: 모든 출력에 CJK 누수 0 검사(최근 버그 재발 방지).
- **CI 연결**: GitHub Actions에서 골든셋 통과를 머지 게이트로. 지표가 목표 미달이면 실패.

이로써 "개선했다"를 느낌이 아니라 숫자로 증명한다.

## 7. 안전 · 신뢰 규격 (자동차 도메인 필수)

안전이 걸린 A/S에서 틀린 단정은 치명적이다. 다음을 규격으로 못 박는다.

- **답변 신뢰도 등급** — 근거 수·검색 점수·모델 확신도로 High/Med/Low 산출.
- **저신뢰 폴백** — Low면 답을 단정하지 말고 "확실치 않음 — 매뉴얼 원문/시니어 확인 권장"으로 유도하고 근거만 제시. (현재 비답변 사후가드를 이 규격으로 승격.)
- **치명 건 하드룰** — 사망·화재·에어백 미전개 신호는 LLM 판단과 무관하게 최상단 고정(트리아지 밴드 90+). [AS-OPERATIONS.md](AS-OPERATIONS.md)의 중요도 밴드와 정합.
- **언어 가드** — 한국어 외 문자 출력 금지(시스템 프롬프트 + 사후 스크럽, 요약은 강한 모델 사용).
- **파괴적 액션 인증** — 문서 삭제·업로드 등은 인증/역할 확인 뒤에만. 현재 무인증 개방 상태를 닫는다.
- **PII 마스킹·감사 로그** — 모든 LLM 호출 기록([GOVERNANCE.md](GOVERNANCE.md)), H-Chat 게이트웨이 정합 유지.

## 8. Agent 요구사항 (thin → 진짜 에이전트)

지금은 도구 2개(RAG/SQL) 라우팅 수준이다. wedge를 진짜 "AI Agent"로 만들려면:

- **다단계 오케스트레이션** — 트리아지 → 진단 → 견적을 계획하고 순차 실행, 각 단계 트레이스.
- **도구 세트** — RAG(매뉴얼), SQL(리콜/불만/부품), 견적 계산(결정적 Java), 체크리스트 매핑.
- **실패 복구** — SQL 자기수정 + 저신뢰 시 재시도/폴백.
- **사람 확인(HITL)** — 견적·발주 같은 액션은 정비사 승인 전 자동 실행 금지.

## 9. 비기능 요구

- **성능**: compute-once 캐시(답변/SQL/요약) 유지, 반복 질의 저지연([GUIDE-answer-cache.md](GUIDE-answer-cache.md)).
- **온디바이스 지향**: 실차 타깃은 미구현이나, **메모리·모델·지연 예산**을 명시(엣지 타깃 가정: Jetson/Android Automotive급). 이 예산이 "온디바이스 최적화" 서사의 근거.
- **데이터 정합**: 불만(DuckDB)·메타(H2)·임베딩(pgvector) 분산으로 인한 JOIN/정렬 한계를 명시하고, 통합 데이터 모델을 P2 과제로.

## 10. 마일스톤

- **P0 — 뼈대 (이 PRD의 본체)**: wedge 흐름 하나로 묶기 · 성공지표+골든셋+CI 게이트 · 저신뢰 폴백 + 치명 하드룰 + 파괴적 엔드포인트 인증.
- **P1 — Agent 고도화**: 다단계 오케스트레이션·트레이스 UI·HITL·실패 복구.
- **P2 — 확장**: 데이터 통합 모델 · 한국 도메인 데이터 교체 · 클라우드/호스티드 LLM 배포 · 피드백→DPO 재학습 루프.

## 11. 리스크 / 미해결

- **데이터 도메인 미스매치** — 현재 코퍼스는 미국 NHTSA(영문)·오너스 매뉴얼. 한국 현대차 A/S 현실과 다름. 프로덕션은 라이선스된 정비 자료·한국 데이터 필요.
- **소형 FT 모델 신뢰도** — 1.5B는 환각·언어 누수 위험. 요약·번역은 강한 모델로, 도메인 응답만 FT로 역할 분리.
- **실차 온디바이스 부재** — "온디바이스"는 현재 개념 증명(맥). 실 타깃 벤치는 로드맵.

### 한 줄 요약

정비사 한 명이 접수 한 건을 **3분 안에 안전하게** 몰고 가게 만드는 것 — 그리고 그 "안전하게"를 **숫자와 폴백으로 증명**하는 것. 이 두 가지가 제품의 뼈대이고, 기존 기능들은 그 밑에 줄 세운다.

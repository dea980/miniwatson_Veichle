# 비전: 온디바이스 차량 음성 어시스턴트 (Voice + On-Device)

> 프로젝트 북극성(North Star). 흩어진 요소(파인튜닝, 양자화, RAG, 거버넌스)를
> **"차 안에서 도는 프라이빗 정비 어시스턴트"** 한 문장으로 통일.
> 관련: [HYUNDAI_NEEDS_ROADMAP.md](HYUNDAI_NEEDS_ROADMAP.md) · [INFERENCE_OPTIMIZATION.md](INFERENCE_OPTIMIZATION.md)

---

## 1. 비전 한 줄

> **"헤이 현대" → 음성으로 정비/리콜 질문 → 온디바이스 파인튜닝 LLM이 매뉴얼 근거(RAG)로 답 → 음성으로 응답. 전부 오프라인이며 프라이빗.**

→ 현대차 실제 방향(Cerence 차량 음성, SDV, H-Chat 데이터 비유출)과 정합.

## 2. 왜 이 비전이 강한가

- **서사 통일**: LoRA(말투) + 양자화(온디바이스) + RAG(근거) + 거버넌스(프라이버시) 가 하나의 사용 시나리오로 묶임.
- **현대차 정합**: 차량 음성 어시스턴트 = 실제 제품 방향. "A·S 정비 질문을 음성으로"는 밸류체인에 직결.
- **온디바이스 = 프라이버시**: 음성과 질의가 기기 밖으로 안 나감 → H-Chat의 "데이터 유출 방지"와 같은 철학.

## 3. 음성 4단계 로드맵 (증분, 각자 독립)

| 단계 | 기술 | 상태 | 특성 |
|---|---|---|---|
| **1. 브라우저 음성** | Web Speech API (STT+TTS) | ✅ 구현 | 쉬움. STT는 클라우드(구글) 경유 |
| **2. 로컬 STT** | Whisper (small/base, 오프라인) | ⬜ | 음성이 기기 밖으로 안 나감(프라이버시) |
| **3. 로컬 TTS** | Piper (경량 오프라인) | ⬜ | 완전 오프라인 응답 |
| **4. 웨이크워드** | openWakeWord / Porcupine ("헤이 현대") | ⬜ | 핸즈프리. 실차 어시스턴트의 핵심 |

> 1→4로 갈수록 "진짜 온디바이스"에 가까워지나, 메모리와 구현 비용↑. M2 단일 기기에선 직렬화 필요.

## 4. 풀 파이프라인 (목표 아키텍처)

```
🎤 음성 입력
  └ [웨이크워드] "헤이 현대" 감지 (always-on 경량 모델)
      └ [STT] Whisper 로컬 → 텍스트
          └ [RAG] 매뉴얼 검색 + 리콜 SQL (근거)
              └ [생성] 파인튜닝 Qwen2.5-1.5B (Q4, 온디바이스) → 답변
                  └ [거버넌스] PII 마스킹·감사 로그
                      └ [TTS] Piper 로컬 → 🔊 음성 응답
```

모든 단계가 **기기 안에서** (클라우드 0) → 프라이빗하고 오프라인으로 도는 차량 어시스턴트.

## 5. 우선순위 (현실)

이 비전은 **정체성/서사**로 선언하되, *빌드*는 핵심부터:

1. **핵심(먼저)**: LoRA 품질 → 양자화 → RAG base vs FT → 거버넌스 데모 (JD 실체)
2. **음성 증분**: 브라우저(✅) → 로컬 Whisper STT → Piper TTS
3. **마지막(선택)**: 웨이크워드 "헤이 현대" (효과 大, 비용 大)

> 음성/웨이크워드는 "위에 얹는 멋진 레이어". 알맹이(파인튜닝, RAG, 거버넌스)가 탄탄해야 빛난다.

## 6. 면접 한 줄

> "브라우저 음성은 오디오가 클라우드로 나가서, 차량과 프라이버시 환경엔 **Whisper 로컬 STT + Piper TTS**가 맞다. 웨이크워드까지 더하면 핸즈프리 온디바이스 어시스턴트가 된다. 즉 음성을 '클라우드 편의'가 아니라 '온디바이스 프라이버시' 관점에서 설계한다."

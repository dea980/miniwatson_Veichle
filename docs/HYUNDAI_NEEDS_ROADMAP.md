## 2. 우리 프로젝트 ↔ 현대차 실제 니즈 매핑

| 현대차가 실제로 필요한 것 | 이 프로젝트가 보여주는 것 | 상태 |
|---|---|---|
| **보안 LLM 게이트웨이**(H-Chat류): PII 차단, 감사, 멀티모델 | 거버넌스 레이어(PII 마스킹·감사로그·provenance) + `LlmClient` 멀티프로바이더 | 🟢 **강한 정합** |
| **자체 도메인 LLM**(Nemotron류): 도메인 데이터로 파인튜닝 | LoRA(PEFT)로 자동차 매뉴얼 도메인화 | 🔄 |
| **온디바이스/차량 추론**(Cerence·SDV): 경량·저지연 | GGUF Q4 양자화 + tok/s 벤치 (메모리1/2·속도2배) | 🟢 |

| **제조/품질·예측정비**: 결함·리콜 분석 | 리콜/DTC text-to-SQL 집계 | 🟢 |

> 핵심 메시지(면접용): "JD 키워드를 따라간 게 아니라, **현대차의 실제 AI 스택(H-Chat 거버넌스 게이트웨이 + 자체 도메인 LLM + 온디바이스 차량 추론 + A/S/품질)을 작게 재현**했다."

## 3. 가장 강한 차별점 — 거버넌스 게이트웨이

대부분의 RAG 사이드 프로젝트엔 없는 게 **거버넌스**다. 그런데 현대차의 H-Chat이 바로 그것(보안·유출방지·감사·멀티모델). 이 프로젝트는:
- **PII 마스킹** (저장 전 민감정보 차단) = 유출 방지
- **감사 로그** (모든 LLM 호출 기록) = 추적성
- **멀티프로바이더 추상화** (ollama/watsonx/…) = 모델 라우팅·교체
→ **H-Chat의 미니 버전**. 이게 이 프로젝트의 가장 현대차다운 부분이고, 면접에서 제일 먼저 꺼낼 카드.

## 4. 니즈 기반 로드맵 (재정렬)

JD 항목이 아니라 **현대차 실제 니즈 우선순위**로 재배치:

**P1 — 이미 강한 것, 마무리만 (현대차 정합 高)**
- [x] 거버넌스 게이트웨이 (PII·감사·멀티프로바이더) — H-Chat 정합
- [x] A/S 매뉴얼 RAG (근거 grounding)
- [x] 리콜/DTC text-to-SQL (품질·결함 분석)
- [x] 온디바이스 경량화 (Q4 양자화 + 벤치)

**P2 — 핵심 보강 (자체 도메인 LLM)**
- [ ] LoRA 파인튜닝 **품질 완성** (깨끗한 데이터 → base vs FT) — Nemotron류 "자체 도메인 LLM" 정합
- [ ] eval 정량화 (도메인 golden recall, judge) — "기술 검증"

**P3 — Agentic (업무 자동화)**
- [ ] AgentController: 질문 → RAG/리콜SQL 툴 선택 → 종합 (Agentic Search)
- [ ] Agent 트레이스 UI (툴 선택 과정 시각화)

**P4 — 데모·배포·서사**
- [ ] 보안 게이트웨이 데모 강조 (PII 마스킹 before/after 화면)
- [ ] 배포 (Vercel + Cloud Run, `llm.provider` 스왑 = 온디바이스↔클라우드)
- [ ] README에 "현대차 실제 스택 매핑" 서사 + 결과 수치

## 5. UI에 꼭 있어야 할 것 (니즈 관점)

현대차 맥락에서 UI가 보여줘야 할 것:
1. **거버넌스 가시성** — PII 마스킹 전/후, 감사 로그(있음), 어떤 모델이 답했는지 → H-Chat류 신뢰
2. **근거(grounding)** — 답변마다 출처 청크 + 점수 (검증가능성)
3. **모델/검색 실험** — base vs FT, rerank/hybrid 비교 (기술검증)
4. **품질·리콜 시각화** — 결함 데이터 차트 (제조/품질 의사결정)

→ "Experiment 탭(모델 A/B + 검색노브)"과 "거버넌스 가시성 강화"가 현대차 니즈에 가장 부합.
